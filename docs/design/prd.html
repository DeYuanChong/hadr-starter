Problem Statement

Before anyone can send aid or forces after a disaster, someone has to correctly answer a smaller question first: what just happened, where, how bad, and who does it affect? For Southeast Asia, that first-hour picture is scattered across three public feeds — GDACS, USGS, and ReliefWeb — that disagree with each other by design: they are three different epistemic objects (a measurement, two independent impact forecasts, and a slow editorial verdict), not three views of the same fact. They also revise, escalate, de-escalate, and delete events after publication, and go down or degrade without warning.

A human decision-maker (or a downstream agent) in the region currently has to read all three feeds themselves every morning, mentally deduplicate the same physical event reported three times under three identifiers, judge which severity signal to trust, and notice on their own when a feed went quiet — with no way to tell whether quiet means "nothing happened" or "the feed is broken."

Solution

An unattended monitoring agent that watches GDACS, USGS, and ReliefWeb on a schedule overnight, keeps only events affecting Southeast Asia, reconciles the same physical event across feeds into one Story, tracks each story through an explicit state machine (new / escalated / de-escalated / revised / deleted / confirmed) instead of treating "new" as the only signal worth acting on, and publishes one situation report to dashboard.html at 08:30 Singapore time. The report always states feed health explicitly — a bad morning is disclosed, never inferred from silence — and always states what changed since the previous report, so a correction or de-escalation is never a silent edit.

The product sits at the sensing layer only. It watches, reconciles, and reports; it does not dispatch aid or recommend a response.

Scope

Geographic scope

Southeast Asia = the ASEAN ten (Brunei, Cambodia, Indonesia, Laos, Malaysia, Myanmar, Philippines, Singapore, Thailand, Vietnam) plus Timor-Leste. Papua New Guinea and southern China are explicitly excluded despite bordering in-scope countries — proximity alone doesn't confer scope (ADR-0001).

Feeds without a country field (USGS points, GDACS cyclone tracks) are filtered with a bounding box, 92°E–141°E, 15°S–29°N, covering the full Sunda Trench to the south and all of Myanmar (including Kachin State) to the north (ADR-0002). A multi-country event is reported in full, including out-of-scope countries it also affects (ADR-0003).

Hazard scope

In scopeOut of scope
Earthquake, tsunami-hint flag, tropical cyclone, flood, volcano, drought, wildfire — the seven hazards GDACS covers. Conflict, complex emergencies, epidemics, heatwaves, landslides. Structurally absent from all three source feeds — the product states this as scope, not failure.

"Multi-hazard" does not mean uniform freshness: real-time for EQ/TC, daily for wildfire/volcano, ~monthly for drought, human-curated for flood. The sitrep discloses this explicitly rather than implying one cadence for all seven hazards (ADR-0017).

Domain language

The full glossary lives in CONTEXT.md. Key terms used throughout this document:

Story

The reconciled representation of one physical disaster. Earthquake stories may merge GDACS + USGS + ReliefWeb records via the solved EQ join. Every other hazard type is sourced solely from GDACS.

Triage severity

The value that decides a story's placement and suppression: for earthquakes, the higher of GDACS's alert colour and USGS PAGER; for every other hazard, the GDACS alert colour alone. Both raw values are always shown — triage severity never hides disagreement, it only decides ordering.

Story state

new escalated / de-escalated revised deleted confirmed (EQ only) — every story is in exactly one state per poll.

User Stories

  1. As a regional decision-maker, I want one morning report covering all Southeast Asian disaster activity, so that I don't have to read three global feeds myself.
  2. As a regional decision-maker, I want the report to state each feed's health up front, so that I can tell a quiet morning from a broken pipeline.
  3. As a regional decision-maker, I want a static disclosure of how often each hazard type actually updates, so that a missing drought entry doesn't read as "no drought risk."
  4. As a regional decision-maker, I want the same physical earthquake reported once, not three times under three different identifiers, so that I'm not double-counting disasters.
  5. As a regional decision-maker, I want to see both GDACS's alert colour and USGS's PAGER forecast when they disagree, so that I can see the actual uncertainty instead of a false consensus.
  6. As a regional decision-maker, I want Green-tier earthquakes suppressed from the report by default, so that constant background seismicity in Indonesia doesn't bury real signal.
  7. As a regional decision-maker, I want an earthquake that escalates out of Green to appear immediately, so that suppression never hides a worsening situation.
  8. As a regional decision-maker, I want an explicit "since yesterday" section, so that I know what changed rather than having to diff today's report against yesterday's myself.
  9. As a regional decision-maker, I want a deleted event to be explicitly flagged as withdrawn once, so that I know to disregard it rather than have it silently vanish.
  10. As a regional decision-maker, I want a cyclone that also affects a non-SEA country shown in full, so that I have the storm's true footprint, not an artificially clipped view.
  11. As a regional decision-maker, I want offshore earthquakes near the Sunda Trench included even though their epicentre isn't on land, so that a real coastal threat isn't silently dropped.
  12. As a regional decision-maker, I want to know when a ReliefWeb page eventually confirms an earthquake I was already told about, so that I have the added editorial context without it re-triggering alarm on an already-assessed event.
  13. As a regional decision-maker, I want any ReliefWeb-derived text to be a short own-words summary with a link back, not a republished excerpt, so that the report respects ReliefWeb's content licensing.
  14. As a regional decision-maker, I want the report published at a predictable time every morning, so that I can build it into my own routine.
  15. As a regional decision-maker, I want to understand that a high-severity event at 14:00 will appear in tomorrow's report rather than trigger a separate alert, so that I know `dashboard.html` is the one place I need to check.
  16. As a downstream agent consuming this report, I want the same structured story data a human sees, so that I can build automated triage on top of it without re-deriving reconciliation logic myself.
  17. As the operator of this system, I want per-feed poll cadences and failures isolated from each other, so that GDACS's known flakiness never blocks USGS or ReliefWeb from reporting.
  18. As the operator of this system, I want a catch-up cursor per feed, so that a period of downtime doesn't cause the next successful poll to silently skip everything that happened while it was down.
  19. As the operator of this system, I want the whole pipeline built without requiring ReliefWeb's appname approval to land in time, so that the 3-day build isn't blocked on an external approval process outside my control.
  20. As the operator of this system, I want persisted state to be a single, human-readable file, so that I can debug an overnight run by opening it directly the next morning.

Implementation Decisions

Reconciliation & state

  • Earthquakes join across feeds via GDACS sourceid ↔ USGS ids list (not the single preferred id, which can change network prefix).
  • All other hazard types (cyclone, flood, volcano, drought, wildfire) are sourced solely from GDACS in v1 — no GDACS↔ReliefWeb reconciliation is attempted. GLIDE is too sparse and non-unique to serve as a join key (ADR-0004). A ReliefWeb page may be attached as a Supplementary link via a loose match (same country, same hazard type, date.created within ~14 days of the story's fromdate) — never merged, never affecting severity.
  • Every story carries one of six states: new, escalated, de-escalated, revised, deleted, confirmed (earthquake stories only, fired when a ReliefWeb page appears via the EQ join) (ADR-0005). Confirmation is purely additive — link and label only, never a severity or suppression change (ADR-0009).
  • Every sitrep carries a "Since yesterday" section listing escalation, de-escalation, revision, confirmation, and deletion of previously reported stories. A deleted story is mentioned exactly once, on the report immediately after deletion, then omitted (ADR-0006).

Severity triage

  • GDACS alert colour and USGS PAGER are both displayed; neither overrides the other. Triage severity — the value used for sorting and suppression — is the higher of the two for earthquakes, or the GDACS alert colour alone for every other hazard (ADR-0007).
  • Stories at Green triage severity (or no alert at all) are suppressed from the sitrep but remain tracked internally. An escalation out of Green is never suppressed (ADR-0008).

Feed operations & persistence

  • No long-running daemon. A scheduled "poll" tick fires every 5 minutes: USGS every tick, GDACS every 2nd tick (~10 min), ReliefWeb every 12th tick (~hourly). A separate "publish" tick fires once at 08:30 SGT (ADR-0010).
  • Each feed has its own catch-up cursor (GDACS datemodified, USGS FDSN updatedafter, ReliefWeb date.created), advanced only after a tick succeeds — a failed tick leaves the cursor unmoved so the next success catches up (ADR-0011).
  • All state — per-story records and per-feed cursors — persists in a single atomically-written JSON file, not a database (ADR-0012).

ReliefWeb build strategy

  • Test the RSS feed with a realistic browser User-Agent first. Build the client behind a small adapter interface so the backing source (fixture / RSS / approved API) is a config swap. Use local fixtures for development only if both RSS and the API are blocked. The live pipeline reports "ReliefWeb: unavailable" honestly rather than substituting fixture data in production (ADR-0013).

Report / dashboard

  • Fixed layout: header → feed health strip (including the static per-hazard cadence disclosure) → since-yesterday → current stories grouped by hazard type, sorted by triage severity descending (ADR-0014, ADR-0017).
  • Any ReliefWeb-derived text is a one-sentence own-words summary plus attribution and a link back — zero direct quotes (ADR-0015).
  • dashboard.html is written only by the 08:30 SGT publish run. No interrupt-driven or out-of-band alerting exists in v1 (ADR-0016).

Testing Decisions

A good test here exercises externally observable behaviour — the story a poll/publish cycle produces — not internal call sequences, since the reconciliation and triage logic is where correctness actually matters and where regressions are least visible by eye.

  • Feed adapters (GDACS / USGS / ReliefWeb fetch + normalize) are tested against fixture responses shaped from the real examples already captured in feeds/*.md and blindspots.md — including the documented edge cases: GDACS NULL-byte / slow responses, USGS 409-on-deleted, ReliefWeb 403 to non-browser clients. No test depends on a live network call.
  • The EQ join is tested as a pure function: given a GDACS record with a sourceid and a set of USGS records, it must match on any entry in ids, not just the preferred id.
  • The state machine is tested table-driven: for each pair of (previous story snapshot, new feed data), assert the resulting state (new/escalated/de-escalated/revised/deleted/confirmed) matches the rules in ADR-0005, independent of any I/O.
  • Triage severity and suppression are tested across the full alert-tier matrix (Green/Orange/Red × GDACS/PAGER present-or-absent) to confirm the max-of-available- signals rule and that an escalation out of Green is never suppressed.
  • The cursor/catch-up logic is tested by simulating a failed tick followed by a successful one, asserting the cursor did not advance on failure and the successful tick's query used the pre-failure cursor.
  • Report rendering is tested by snapshotting dashboard.html output against a fixed persisted-state fixture, covering: an all-feeds-healthy morning, a GDACS-degraded morning, and a morning with a since-yesterday deletion.

Out of Scope

  • Geography: anything outside ASEAN-10 + Timor-Leste, including Papua New Guinea and southern China, regardless of severity.
  • Hazards: conflict, complex emergencies, epidemics, heatwaves, landslides — structurally absent from all three source feeds.
  • Tsunami warnings: USGS's tsunami flag is surfaced as a "check NOAA" hint only; actual tsunami warnings are NOAA's job, not this product's.
  • Response / dispatch: this product senses, reconciles, and reports — it never recommends or triggers a response action.
  • Cross-feed reconciliation for non-earthquake hazards: GDACS is sole authority for cyclone/flood/volcano/drought/wildfire in v1; no ReliefWeb merge is attempted for them.
  • Out-of-band / real-time alerting: no notification channel exists besides the scheduled 08:30 dashboard.html publish.
  • Quoting ReliefWeb content: zero direct quotes under any circumstance, regardless of newsworthiness.
  • An "elsewhere in the world" footnote: considered and rejected as inconsistent with the project's hard scope line.
  • A database: persistence is a single JSON file for this build; a database is not planned unless data volume outgrows it.

Further Notes

This PRD was produced via a structured grilling session against REQS.md, the three feed docs, and blindspots.md — the full question-by-question trail is in QUESTIONS.md, the resolved domain vocabulary in CONTEXT.md, and the rationale behind every hard-to-reverse or surprising decision in the 17 ADRs indexed below.

An A2 consistency pass across CONTEXT.md and all ADRs found and fixed four issues before this PRD was written — most notably that the original SEA bounding box (21°N northern limit) would have clipped part of in-scope Myanmar and silently broken the GDACS↔USGS earthquake join for northern-Myanmar events. This is recorded in QUESTIONS.md under "A2 — Inconsistency check."

Given the 3-day build window, the first vertical slice matters more than completeness: the earthquake path (solved cross-feed join, dual-severity triage, full state machine) is the highest-value slice to build and demo first, since it's the only hazard type with a genuine three-feed reconciliation story. The other five GDACS hazard types share the same suppression/state/report machinery but with a simpler single-source data path, so they should follow naturally once the earthquake slice works end-to-end.
